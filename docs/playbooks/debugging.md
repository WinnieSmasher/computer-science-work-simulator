# 调试与排障 Playbook

这份 playbook 用来训练真实工作里最常见的一类任务：别人告诉你“功能不对”，但没有告诉你原因。

## 1. 复现

- 写下用户原话，不急着改代码。
- 把输入、账号角色、接口地址、时间、期望结果和实际结果记录清楚。
- 先用测试或 curl 复现，再打开 IDE debug。

## 2. 观察

- 看 HTTP 状态码、返回体里的 `code/message/data`。
- 看 Controller 收到的参数是不是你以为的值。
- 看 Service 里有没有 trim、大小写、默认值、状态过滤、分页边界。
- 看 Repository 或数据库查询结果是否包含目标数据。

## 3. 定位

常见根因优先级：

- 输入归一化：大小写、首尾空格、空字符串。
- 状态过滤：`ACTIVE/DRAFT`、逻辑删除、上下架时间。
- 分页边界：第一页从 0 还是 1 开始，`hasMore/isEnd` 是否一致。
- 数据问题：脏数据、旧字段、重复数据、缺少索引。
- 环境问题：本地、测试、生产数据源不一致。

## 4. 修复

- 修最小范围，不顺手重构整条链路。
- 让用户端接口保持“只返回可展示数据”的约束。
- 如需排查不可见数据，新增后台诊断或后台列表能力，不污染用户端接口。

## 5. 回归

- 测试名要描述 bug 场景。
- 覆盖正常输入、大小写、空格、草稿、逻辑删除。
- 在 PR 或提交说明里写清楚根因，不只写“fix bug”。
